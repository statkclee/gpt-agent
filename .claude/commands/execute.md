# /execute - 챕터 내 코드 실행 및 검증

대상: $ARGUMENTS

(파일명 지정, "all"로 전체, "dry-run"으로 실행 없이 점검만)

## 목적

챕터에 포함된 코드 블록이 실제로 동작하는지 검증한다. 독자가 따라 했을 때 오류 없이 실행되어야 책의 신뢰도가 유지된다.

## 작업 절차

### 1단계: 코드 블록 추출

대상 `.qmd` 파일에서 실행 가능한 코드 블록을 식별한다.

**추출 대상:**
- ````{python}` — Python 코드
- ````{r}` — R 코드
- ````{bash}` / ````{shell}` — 쉘 명령어
- ````python` / ````r` / ````bash` — 실행 가능하지만 Quarto 렌더링 대상 아닌 코드

**제외 대상:**
- ````yaml`, ````json`, ````toml` — 설정 파일 (실행 불가)
- ````{python} #| eval: false` — 명시적 실행 제외 표시
- ````text`, ````markdown` — 출력 예시
- 의사 코드(pseudocode)로 표시된 블록

### 2단계: 의존성 확인

코드 실행 전 필요한 패키지와 환경을 점검한다.

**Python:**
- `requirements.txt` 대비 import 문 확인
- 설치되지 않은 패키지 목록 생성
- Python 버전 호환성 확인

**R:**
- `renv.lock` 대비 library() 호출 확인
- 미설치 패키지 목록 생성

**시스템:**
- CLI 도구 존재 여부 (docker, quarto, git 등)
- API 키 필요 여부 (환경변수 점검)

### 3단계: 코드 실행

각 코드 블록을 순서대로 실행하며 결과를 기록한다.

**실행 규칙:**
- 챕터 내 코드는 위에서 아래로 순차 실행 (상태 누적)
- API 호출, 외부 서비스 접근 코드는 실행하지 않고 표시만 한다
- 파일 시스템 변경(쓰기, 삭제) 코드는 임시 디렉토리에서 실행한다
- 실행 시간이 30초를 초과하면 타임아웃 처리한다

**dry-run 모드:**
- 코드를 실행하지 않고 문법 오류, import 누락, 변수 참조 오류만 정적 분석한다

### 4단계: 결과 검증

코드 실행 결과를 챕터에 기록된 예상 출력과 비교한다.

**검증 항목:**
- 실행 성공/실패 여부
- 출력 결과가 챕터 설명과 일치하는가
- 경고(warning) 발생 여부
- deprecated API 사용 여부

## 출력 형식

```

코드 블록 현황: 총 X개 (Python X, R X, Bash X, 제외 X)

■ 의존성 점검

  ⚠️ R 패키지: ggplot2 미설치

## 자동 수정 제안

실패하거나 경고가 발생한 코드에 대해 수정안을 제시한다.

**수정 가능한 문제:**
- deprecated API → 최신 API로 변환
- import 오류 → 올바른 패키지명/경로 제안
- 문법 오류 → 수정 코드 제시
- 버전 불일치 → requirements.txt/renv.lock 업데이트 제안

**수정 불가 (보고만):**
- 외부 서비스 의존 코드 (API 키, 네트워크)
- 하드웨어 의존 코드 (GPU, 특정 OS)
- 데이터 파일 의존 코드 (사용자 데이터셋)

## 주의사항
- 코드 실행은 현재 환경에 영향을 줄 수 있으므로 신중하게 진행한다.
- API 키가 필요한 코드는 절대 실행하지 않는다 (보안).
- 파일 쓰기/삭제 코드는 임시 디렉토리에서만 실행한다.
- dry-run 모드를 먼저 실행하여 안전성을 확인한 뒤 전체 실행을 권장한다.
- 실행 결과로 챕터 수정이 필요하면 `/write-chapter`로 이어서 작업한다.
- 의존성 문제 발견 시 `requirements.txt` 또는 `renv.lock` 업데이트를 제안한다.
