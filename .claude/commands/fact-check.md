# /fact-check - 기술 사실 검증

대상: $ARGUMENTS

(파일명 지정, "all"로 전체 검증)

## 목적

챕터에 기술된 기술 정보가 정확하고 최신인지 검증한다. API 버전, 도구 이름, URL 유효성, 코드 문법, 통계/수치를 점검하여 책의 신뢰도를 보장한다.

## 검증 항목

### 1. API/라이브러리 버전 (심각도: 높음)

기술서에서 가장 빠르게 구버전이 되는 정보.

**점검 대상:**
- Python 패키지 버전 (openai, langchain, anthropic 등)
- API 호출 방식 (deprecated 여부)
- 함수/메서드 시그니처 변경 여부
- 설정 파라미터 변경 여부

**검증 방법:**
- 패키지 공식 문서/changelog 확인
- `pip show [패키지]`로 현재 설치 버전 확인
- 코드 내 API 호출이 최신 문법인지 확인

### 2. 도구/서비스 정보 (심각도: 높음)

AI 도구 생태계는 변화가 빠르다.

**점검 대상:**
- 도구 이름 변경 (예: GitHub Copilot → GitHub Copilot 변경사항)
- 가격 정책 변경
- 기능 추가/제거
- 서비스 종료 여부
- 라이선스 변경

**검증 방법:**
- 공식 웹사이트에서 최신 정보 확인
- 웹 검색으로 최근 변경사항 조사

### 3. URL 유효성 (심각도: 중간)

깨진 링크는 독자 경험을 크게 해친다.

**점검 대상:**
- 본문 내 하이퍼링크
- 참고 문헌 URL
- 이미지 외부 소스 URL
- 코드 예제 내 URL

**검증 방법:**
- HTTP HEAD 요청으로 응답 코드 확인
- 리다이렉트 발생 시 최종 URL 보고
- 404 등 오류 시 대체 URL 제안

### 4. 코드 정확성 (심각도: 중간)

코드가 설명과 일치하고 현재 환경에서 동작하는지.

**점검 대상:**
- 문법 오류 (syntax error)
- 미정의 변수/함수 참조
- import 경로 정확성
- 출력 결과가 설명과 일치하는지

**검증 방법:**
- 정적 분석 (문법, 타입)
- 실제 실행 검증은 `/execute` 스킬로 위임

### 5. 수치/통계 (심각도: 낮음)

본문에 인용된 숫자가 정확한지.

**점검 대상:**
- 모델 파라미터 수 (예: "GPT-4는 X개의 파라미터")
- 벤치마크 점수
- 시장 점유율, 사용자 수
- 날짜, 출시일

**검증 방법:**
- 공식 발표/논문 대조
- 명확한 출처가 없는 수치는 플래그

## 출력 형식

```
🔎 사실 검증: [파일명]

■ API/라이브러리
  [높음] 라인 45: openai.ChatCompletion.create()
    → 상태: ❌ deprecated (v1.0+에서 제거)
    → 최신: client.chat.completions.create()
    → 수정 코드 제안 포함

  [높음] 라인 88: langchain.llms.OpenAI
    → 상태: ⚠️ 경로 변경 (langchain-openai 패키지로 분리)
    → 최신: from langchain_openai import OpenAI

■ 도구/서비스
  [높음] 라인 120: "Cursor는 VS Code 포크 기반"
    → 상태: ✅ 정확 (2026년 기준)
  
  [중간] 라인 155: "GitHub Copilot 월 $10"
    → 상태: ⚠️ 가격 변경 확인 필요
    → 최신 가격: 웹 검색 결과 참조

■ URL
  [중간] 라인 200: https://example.com/old-docs
    → 상태: ❌ 404 Not Found
    → 제안: https://example.com/new-docs (리다이렉트 목적지)

  [낮음] 라인 230: https://arxiv.org/abs/2303.xxxxx
    → 상태: ✅ 유효

■ 코드
  [중간] 라인 90: `model="gpt-4"` 파라미터
    → 상태: ⚠️ 동작하나 최신 모델은 "gpt-4o"
    → 제안: 최신 모델명 업데이트 고려

■ 수치/통계
  [낮음] 라인 30: "GPT-4는 약 1.8조 개의 파라미터"
    → 상태: ⚠️ 비공식 추정치 — 출처 명시 권장

━━━━━━━━━━━━━━━━━━━━━━━━
요약: ❌ 오류 X건, ⚠️ 확인필요 X건, ✅ 정확 X건
신뢰도: [높음/보통/낮음]
```

## 작업 절차

1. 대상 `.qmd` 파일을 전체 읽는다.
2. 검증 대상 항목을 카테고리별로 추출한다.
3. 코드 블록 내 API 호출, import 문을 분석한다.
4. URL이 있으면 유효성을 확인한다.
5. 웹 검색으로 도구/서비스 최신 정보를 대조한다.
6. 결과를 심각도 순으로 정렬하여 보고한다.
7. 자동 수정 가능한 항목은 수정안을 제시한다.

## 주의사항
- URL 검증 시 과도한 요청을 보내지 않는다 (rate limiting 준수).
- API 키가 필요한 외부 서비스 호출은 하지 않는다.
- "정확하지 않을 수 있음" 수준의 불확실한 검증 결과는 명시적으로 표시한다.
- 검증 시점(날짜)을 기록하여, 이후 재검증 시 참고한다.
- 수정 작업은 사용자 확인 후 `/write-chapter`로 진행한다.
- 코드 실행 검증이 필요하면 `/execute`를 권장한다.
