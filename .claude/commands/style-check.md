# /style-check - 문체 검수

대상: $ARGUMENTS

(파일명 지정 또는 "all"로 전체 검사)

## 검사 항목

### 1. "이 + 명사" 패턴 (최우선, 심각도: 높음)
AI 생성 문장의 가장 전형적인 특징. 반드시 탐지하여 수정 제안한다.

**탐지 패턴:**
- "이 메서드", "이 함수", "이 클래스", "이 변수", "이 객체"
- "이 문제", "이 오류", "이 방식", "이 경우", "이 과정"
- "이처럼", "이러한", "이와 같이", "이를 통해"
- "이것은", "이것이", "이것을"

**수정 방향:** 구체적인 명사로 대체
- "이 메서드는" → "`startswith()`는" (실제 메서드명)
- "이 문제를" → "인덱스 오류를" (구체적 문제)
- "이러한 특성" → "시퀀스라는 특성" (구체적 명칭)

### 2. 경어체 혼용 (심각도: 높음)
`index.qmd`를 제외한 모든 파일은 평서문이어야 한다.

**탐지 패턴:**
- "~합니다", "~입니다", "~됩니다", "~습니다"
- "~하세요", "~보세요", "~주세요"
- "~할 수 있습니다", "~해야 합니다"

**수정 방향:**
- "~합니다" → "~한다" / "~함"
- "~입니다" → "~이다" / "~임"

### 3. 기계적 연결어 (심각도: 중간)

**탐지 패턴:**
- 문단 시작: "다음으로,", "또한,", "마지막으로,", "먼저,"
- "추가적으로,", "더불어,", "뿐만 아니라,"
- "결론적으로,", "종합하면,", "요약하면,"

**수정 방향:** 연결어 제거하고 문장 자체로 흐름을 만든다.

### 4. 불필요한 관형사형 (심각도: 중간)

**탐지 패턴:**
- "~하는 것이다", "~하는 것임", "~하는 것을"
- "~할 수 있는 것이다"
- "~라는 점이다", "~라는 것이다"

### 5. 과도한 격식 (심각도: 낮음)

**탐지 패턴:**
- "본 연구에서는", "본 장에서는", "본 절에서는"
- "상기한", "하기의", "전술한"

## 출력 형식

파일별로 위반 사항을 다음 형식으로 보고한다:

```
🔍 문체 검수 결과: [파일명]

[높음] 라인 XX: "이 메서드는 빈 문자열에서도..."
  → 제안: "`startswith()`는 빈 문자열에서도..."

[높음] 라인 XX: "이러한 특성 덕분에..."
  → 제안: "시퀀스라는 특성 덕분에..."

[높음] 라인 XX: "사용할 수 있습니다"
  → 제안: "사용할 수 있다"

[중간] 라인 XX: "또한, 다음과 같은..."
  → 제안: 연결어 제거

━━━━━━━━━━━━━━━━━━━━━━━━
요약: 높음 X건, 중간 X건, 낮음 X건
```

## 작업 절차

1. 대상 파일(들) 읽기
2. 각 검사 항목의 패턴을 줄 단위로 탐지
3. 코드 블록(``` 안), YAML 헤더(--- 안), HTML 태그 안의 내용은 검사에서 제외
4. 위반 사항을 심각도별로 정렬하여 보고
5. 수정을 원하면 사용자 확인 후 자동 수정 실행

## 주의사항
- 코드 주석 내 한글도 검사 대상이다.
- 인용문(> 블록)은 원문 존중으로 검사 제외한다.
- `index.qmd`는 경어체 검사를 건너뛴다 (서문은 존칭 허용).
- 오탐(false positive)이 있을 수 있으므로, 자동 수정 전 반드시 사용자 확인을 받는다.
