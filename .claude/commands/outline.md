# /outline - 챕터 아웃라인 설계/재구성

대상: $ARGUMENTS

(파일명 지정으로 기존 챕터 재구성, "새주제명"으로 신규 아웃라인 설계)

## 목적

집필 전 챕터의 뼈대를 설계한다. 헤딩 계층, 절별 분량 배분, 학습 흐름, 다이어그램 배치 계획을 수립하여 `/write-chapter` 실행 전 구조적 기반을 마련한다.

## 설계 원칙

### 학습 흐름 설계
- **왜(Why)** → **무엇(What)** → **어떻게(How)** → **실전(Practice)** 순서
- 독자가 이미 아는 개념에서 출발하여 새 개념으로 확장
- 각 절이 독립적으로도 참조 가능하되, 순서대로 읽을 때 최적

### 분량 균형
- 한 챕터 목표: 400-800줄 (A4 기준 10-20페이지)
- 한 섹션(##) 목표: 80-150줄
- 한 하위 섹션(###) 목표: 30-80줄
- 텍스트 50-60%, 코드 25-35%, 시각 자료 10-20% 비율 권장

### 시각 요소 배치
- 핵심 개념마다 Tufte style SVG 다이어그램 1개 이상 계획
- 텍스트만 100줄 이상 이어지지 않도록 시각 요소 배치
- 코드 → 설명 → 다이어그램 리듬 유지

## 작업 절차

### 기존 챕터 재구성

1. 대상 `.qmd` 파일을 읽고 현재 헤딩 구조를 추출한다.
2. `PLAN.md`에서 해당 챕터 목표를 확인한다.
3. 같은 부(Part) 내 다른 챕터의 헤딩을 훑어 중복/빈틈을 파악한다.
4. 현재 구조의 문제점을 진단한다.
5. 개선된 아웃라인을 제안한다.

### 신규 아웃라인 설계

1. `PLAN.md`에서 해당 주제의 위치와 목표를 확인한다.
2. 같은 부(Part) 내 기존 챕터와의 관계를 파악한다.
3. 독자 수준과 선행 지식을 고려하여 흐름을 설계한다.
4. 아웃라인을 제안한다.

## 출력 형식

```
📐 아웃라인: [챕터 제목]

■ 위치: 8부 AI 시대 프로그래밍 / 3번째 챕터
■ 선행 챕터: p1-paradigm.qmd (재사용에서 재생성으로)
■ 후행 챕터: p1-workflow.qmd (AI 코딩 실전 워크플로우)
■ 목표 분량: 약 500줄

━━━━━━━━━━━━━━━━━━━━━━━━

## 도입 — 왜 컨텍스트 엔지니어링인가 (~60줄)
   프롬프트의 한계 → 컨텍스트라는 상위 개념 소개
   📊 다이어그램: 프롬프트 vs 컨텍스트 범위 비교

## 컨텍스트의 구성 요소 (~120줄)
   ### 시스템 프롬프트 (~40줄)
   ### 사용자 컨텍스트 (~40줄)
   ### 도구와 외부 지식 (~40줄)
   📊 다이어그램: 컨텍스트 구성 요소 계층도

## 실전 컨텍스트 설계 패턴 (~150줄)
   ### 패턴 1: ... (~50줄)
       💻 코드 예제
   ### 패턴 2: ... (~50줄)
       💻 코드 예제
   ### 패턴 3: ... (~50줄)
       💻 코드 예제

## 도구별 컨텍스트 전략 (~100줄)
   ### Claude Code (~35줄)
   ### Cursor (~35줄)
   ### 비교표 (~30줄)
   📊 다이어그램: 도구별 컨텍스트 흐름

## 정리 (~70줄)
   핵심 요약, 체크리스트, 다음 챕터 연결

━━━━━━━━━━━━━━━━━━━━━━━━
다이어그램 계획: 3개 (SVG, Tufte style grayscale)
코드 예제 계획: 3개 (Python)
예상 분량: ~500줄
```

### 기존 챕터 재구성 시 추가 출력

```
■ 현재 구조 진단
  [문제] 섹션 3, 4가 내용상 하나로 묶여야 함
  [문제] 도입부 없이 바로 기술 설명 시작
  [문제] 다이어그램이 후반부에 편중

■ 변경 사항
  - 섹션 3+4 통합 → "실전 컨텍스트 설계 패턴"
  - 도입부 신규 추가 (~60줄)
  - 다이어그램 2개 → 3개로 확대, 전반부에 1개 추가
  
■ 이동/재배치가 필요한 기존 내용
  - 라인 45-80 → 새 섹션 2로 이동
  - 라인 200-220 → 삭제 (p1-paradigm.qmd와 중복)
```

## 주의사항
- 아웃라인 설계만 수행한다. 본문을 작성하지 않는다.
- 사용자 승인 후 `/write-chapter`로 실제 집필을 진행한다.
- 기존 챕터 재구성 시 내용 삭제가 아닌 재배치를 우선한다.
- `_quarto.yml`의 챕터 순서와 일관성을 유지한다.
