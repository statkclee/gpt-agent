# /review - 챕터 원고 리뷰

대상: $ARGUMENTS

(파일명 지정 또는 "all"로 전체 리뷰)

## 리뷰 관점

### 1. 구조 리뷰 (심각도: 높음)

챕터의 뼈대가 탄탄한지 점검한다.

**점검 항목:**
- 헤딩(##, ###) 계층 구조가 논리적인가
- 도입부에서 챕터의 목표와 맥락을 제시하는가
- 각 절(section)이 자연스러운 흐름으로 연결되는가
- 결론부에서 핵심 내용을 정리하고 다음 챕터로 연결하는가
- 헤딩 깊이가 4단계(####) 이상 남용되지 않았는가

### 2. 논리 리뷰 (심각도: 높음)

주장과 설명의 논리적 일관성을 점검한다.

**점검 항목:**
- 개념 설명 순서가 선행 지식 → 새 개념 순서인가
- 용어를 처음 사용할 때 정의 또는 설명이 있는가
- 주장에 근거(예시, 코드, 인용)가 뒷받침되는가
- 같은 개념을 다른 용어로 혼용하지 않았는가
- 독자 수준에 맞는 설명 깊이인가

### 3. 완성도 리뷰 (심각도: 중간)

빈 부분, 미완성 표시, 품질 저하 구간을 찾는다.

**점검 항목:**
- TODO, FIXME, TBD, WIP 등 미완성 마커
- 빈 섹션 또는 제목만 있는 섹션
- 코드 블록에 설명이 빠진 곳
- 이미지/다이어그램 참조가 깨진 곳
- callout 블록(note, tip, warning)이 적절히 활용되었는가

### 4. 상호 참조 리뷰 (심각도: 중간)

같은 부(Part) 내 다른 챕터와의 관계를 점검한다.

**점검 항목:**
- 다른 챕터와 내용이 중복되는 부분이 있는가
- 관련 챕터로의 교차 참조(`@sec-`)가 적절한가
- 용어 사용이 다른 챕터와 일관적인가
- `glossary.qmd`에 등록해야 할 새 용어가 있는가

### 5. 독자 경험 리뷰 (심각도: 낮음)

독자 입장에서 읽기 좋은지 점검한다.

**점검 항목:**
- 긴 텍스트 블록 사이에 시각 요소(코드, 다이어그램, 표)가 적절히 배치되었는가
- 한 섹션이 지나치게 길지 않은가 (스크롤 피로)
- 실습/예제가 충분한가
- 코드와 설명의 비율이 적절한가

## 출력 형식

```
📖 원고 리뷰: [파일명]

■ 구조
  [높음] 3장 → 4장 전환이 갑작스럽다. 중간에 연결 문단 필요.
  [높음] ### 수준 헤딩이 10개 이상. 일부를 상위 절로 묶을 것을 권장.

■ 논리
  [높음] 라인 45: "컨텍스트 엔지니어링"을 정의 없이 사용.
  [중간] 라인 120-135: RAG 설명이 앞서 나온 프롬프트 엔지니어링과 모순.

■ 완성도
  [중간] 라인 200: TODO 마커 발견 — "예제 코드 추가 필요"
  [중간] 라인 88: `images/workflow-detail.svg` 파일 없음.

■ 상호 참조
  [중간] 72-paradigm-shift.qmd와 "재사용 vs 재생성" 내용 중복 (라인 50-65).

■ 독자 경험
  [낮음] 라인 150-250: 텍스트만 100줄 이상 이어짐. 중간에 다이어그램 삽입 권장.

━━━━━━━━━━━━━━━━━━━━━━━━
요약: 높음 X건, 중간 X건, 낮음 X건
총평: [전체적인 평가와 우선 개선 사항 1-2문장]
```

## 작업 절차

1. `PLAN.md`에서 해당 챕터의 목표와 의도를 파악한다.
2. 대상 `.qmd` 파일을 전체 읽는다.
3. 같은 부(Part)의 다른 챕터 목차(헤딩)를 훑어본다.
4. `images/` 폴더에서 참조 이미지 존재 여부를 확인한다.
5. 5가지 관점으로 리뷰를 수행한다.
6. 심각도 순으로 정렬하여 보고한다.
7. 사용자가 수정을 요청하면 `/write-chapter`로 이어서 작업한다.

## 주의사항
- 리뷰는 읽기 전용 작업이다. 파일을 수정하지 않는다.
- 문체 검수는 `/style-check` 스킬 영역이므로 여기서는 다루지 않는다.
- 코드 실행 검증은 `/execute` 스킬 영역이므로 여기서는 다루지 않는다.
- 리뷰 결과를 바탕으로 `/write-chapter`, `/style-check`, `/execute` 사용을 제안할 수 있다.
